Static site, dynamic product

GitHub Pages is only the landing page. The actual planner runs as a local CLI/TUI, a local MCP server, or the hosted MCP endpoint on Vercel.

Planner model

Dataset

Provider ledger, not vague recommendations

The planner works from a curated CSV with provider class, compute notes, card requirements, API support, and source links.

Heuristics

Compute lane classification

Requests get normalized into burst, session, heavy, or grant-scale lanes based on VRAM, runtime, and job fan-out.

Workflow

Stage-aware planning

The output is not only a ranking. The planner decides what stays local and what should move remote for each stage.

Current heuristics

Lane thresholds

These are the kinds of rules that make the landing page feel technical in a good way: they expose how the product reasons.

grant-scale  => scratch-train
              or VRAM >= 32 GB
              or task_hours >= 18

heavy        => VRAM >= 20 GB
              or task_hours >= 8
              or parallel_jobs >= 4

session      => VRAM >= 12 GB
              or task_hours >= 3
              or parallel_jobs >= 2

Provider lanes

Burst

Short, low-friction work

Best for quick inference, demos, notebooks, and small agent loops that should not require a long-lived allocation.

Session / heavy

Medium runs and repeated evals

This is where credits, starter plans, and API-friendly providers start outperforming pure free-tier options.

Grant-scale

When the compute stops being casual

For larger VRAM and longer jobs, the planner shifts toward programs, allocations, and grant-style infrastructure instead of pretending a notebook queue is enough.
